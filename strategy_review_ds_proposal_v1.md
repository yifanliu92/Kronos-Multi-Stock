## 一票否决：执行分辨率不匹配

**DeepSeek 方案核心假设：** 每 120 秒检查一次持仓，允许秒级强制平仓。

**Kronos 实际执行线：** 每 10 分钟（600 秒）一次 cron 触发。

这是 **5 倍分辨率差距**，不是一个可以妥协的参数。

### 实盘数据验证这个差距意味着什么

基于 90 个连续 10 分钟时隙的 P&L 变化：

| 指标 | 数值 |
|------|------|
| 相邻时隙 P&L 中位数 | 0.00% |
| P&L 变化在 [-0.3%, +0.3%] 内 | **68.9%** |
| P&L 变化 > +0.5% | **8.9%** |
| P&L 变化 < -0.5% | **14.4%** |

**如果按 120 秒执行（线性缩放）：**
- 预期 120 秒 P&L 变化 ≈ -0.009%
- +0.5% 浮盈触发移动止盈：概率极低（<1%）
- -0.8% 浮亏止损：概率同样极低（<1%）
- **结果：绝大多数仓位在 120 秒计时结束时都没有触发任何阈值，纯粹靠时间止损平仓**

**结论：** 120 秒计时器在 Kronos 的 10 分钟执行线上无法工作。但 DeepSeek 方案的**核心思想**（时间作为风控变量）可以用 10 分钟分辨率重新实现。

## 方案优点

虽然执行分辨率有硬伤，方案有两个核心想法确实抓住了当前策略的痛点：

### 1. 时间作为风控变量（√）

当前策略的致命问题：**持仓不变时胜率 0.0%** —— 一旦开仓后行情反转，没有任何主动退出机制，只能硬扛到下一次信号反转。

DeepSeek 提出"到了时间就强制处理"的思路，在 10 分钟槽位上同样有价值：**在没有反转信号的时隙里，不做被动等待，而是主动降低暴露。**

### 2. 非对称止盈止损（√）

当前盈亏比 0.53（盈利平均 +1.13%，亏损平均 -2.14%），确实需要改善。

DeepSeek 的"浮盈小就奔跑、浮亏小就斩仓"是对的，只是需要适配到 10 分钟分辨率。

## 数据验证：为什么早盘禁止是对的

实盘 95 条审查记录的按时段胜率：

| 时段 | 时点数 | 盈利时点 | 时段胜率 | 分析 |
|------|--------|----------|----------|------|
| 09:00-09:59 | 7 | 0 | **0.0%** | 开盘噪声最大 |
| 10:00-10:59 | 18 | 1 | **5.6%** | 方向不稳定 |
| 11:00-11:59 | 13 | 0 | **0.0%** | 临近午盘无方向偏好 |
| **13:00-13:59** | **28** | **9** | **32.1%** | **唯一可盈利时段** |
| 14:00-14:59 | 29 | 0 | **0.0%** | 尾盘方向恶化 |
| 15:00 | 4 | 0 | 0.0% | 收盘快照 |

**数据铁律：全天的盈利窗口只有一个：13:00-14:00。** 剩余时段方向胜率为 0%~5.6%。

## 优化后的 Kronos 可执行方案

### 设计原则

```
将 DeepSeek 的"时间优先 + 不对称持仓"思想，
适配到 10 分钟分辨率下，
用我们已有的 simulate_rules_603305.json 参数体系表达。
```

### 改动量：仅修改 `simulate_rules_603305.json` 的约 5 个参数区块

不新增脚本、不改变执行线、不改信号触发逻辑。

### 核心修改 1：增加时间止损（slot-based）

**当前：** 只在信号反转或止盈触发时才变仓。
**问题：** 开仓后即使方向错了，只要后续三个 10 分钟都是中性，就一直扛着亏损。
**改为：** 开仓后若连续 N 个 10 分钟时隙未出现反转信号 → 强制减仓至底仓。

在 `simulate_rules_603305.json` 中新增一个区块：

```json
"time_based_stop": {
  "enabled": true,
  "_comment": "连续多少个10分钟时隙未出现反转信号，强制减到底仓。1 slots = 10分钟。",
  "max_slots_without_signal_change": 3,
  "reduce_to_pct": 20,
  "exclude_hours": ["13", "14"],
  "_exclude_comment": "13:00-14:59 下午波动窗口，不做时间强制"
}
```

### 核心修改 2：早盘禁止加仓（位置管理）

**当前：** 全天信号一致，早盘波动大照样加仓/加空。
**问题：** 数据证实早盘胜率 0%~5.6%，任何加仓行为都是负期望。
**改为：** 开盘后首 3 个时隙（09:30-10:00）只允许减仓和平仓，不允许增加暴露。

在 `position_management` 中新增：

```json
"morning_cool_down": {
  "enabled": true,
  "_comment": "09:30-10:00 禁止任何加仓动作，仅允许减仓/平仓",
  "start": "0930",
  "end": "1000",
  "actions_allowed": ["reduce_only", "flat"]
}
```

### 核心修改 3：不对称止盈止损（收窄止损、放低止盈起点）

**当前止盈起点 3.0%** → 在过去 22 个交易日里只有 2 次出现（仅占 2.1% 时点）。
**问题：** 止盈起点太高，绝大多数盈利从没触发过止盈就转亏了。
**改为：** 采用影子策略已验证的参数（止盈 1.2%/2.0%/3.0%）。

**当前止损 1.0%/2.0%/3.0%（无保本止盈）** → 但数据表明亏损 69.4% 在 -2% 以内。
**问题：** 没有"浮盈保护"机制。浮盈转浮亏是常态。
**改为：** 增加一条在保本附近自动减仓 50% 的规则。

修改 `take_profit.profit_levels`：

```python
# 当前
profit_pct: 3.0 -> reduce 30%
profit_pct: 5.0 -> reduce 30%
profit_pct: 7.0 -> reduce 20%

# 改为（影子策略已验证）
profit_pct: 1.2 -> reduce 30%   # 新增：低起点止盈，让利润先锁定一部分
profit_pct: 2.0 -> reduce 15%   # 影子参数，已在81%时点优于主策略
profit_pct: 3.0 -> reduce 20%   # 影子参数
```

增补 `drawdown_levels`：

```python
# 当前
drawdown 1.0% -> reduce 30%
drawdown 1.8% -> reduce 30%
drawdown 2.5% -> floor 20%

# 增加一条保本回撤减仓
drawdown 0.3% -> reduce 50%   # 新增：从最高点回撤 0.3% 即大幅减仓
```

### 核心修改 4：下午窗口加重暴露

**当前：** 全天仓位管理一致。
**问题：** 13:00-14:00 胜率 32.1%，这段窗口应允许更高仓位上限。
**改为：** 下午 13:00-14:00 启动"加仓快速通道"。

```json
"afternoon_enhance_window": {
  "enabled": true,
  "start": "1300",
  "end": "1400",
  "_comment": "下午窗口胜率最高，允许更积极的加仓",
  "bull_add_pct_bonus": 10,
  "strong_bull_add_pct_bonus": 15
}
```

### 修复前（当前策略）

|------|------|
| 方向预测胜率 | 48.05% |
| 持仓 P&L 胜率 | **10.5%** |
| 平均盈利 | +1.13% |
| 平均亏损 | -2.14% |
| 盈亏比 | **0.53** |
| 日方向决策数 | ~22 次 |

### 修复后（预期）

| 指标 | 预期改善 | 推演逻辑 |
|------|----------|----------|
| 方向预测胜率 | ~48%（不变） | 不修改信号层 |
| 持仓 P&L 胜率 | **25-35%** | 时间止损退出大部分浮亏持仓 |
| 平均盈利 | +1.0-1.5% | 止盈起点降低后锁利更频繁 |
| 平均亏损 | -0.8-1.5% | 0.3%回撤减仓+时间止损限制亏损 |
| 盈亏比 | **1.0-1.5** | 主要改善项 |
| 日方向决策数 | ~12-15 次 | 早盘禁止加仓自然降低频率 |
| 最大回撤 | 显著降低 | 时间止损防止深度套牢 |

### 关键估算逻辑

当前 13:00-14:00 时段有 32.1% 时点胜率。如果：
- 早盘禁仓（损失 0-5% 胜率的早盘窗口）
- 下午增强（利用 32.1% 时段）
- 其余时段用时间止损保本退出

则预期持仓胜率 ≈ 32% × 0.7（下午占比调整）≈ **22-30%**。

同时盈亏比从 0.53 → **1.0-1.5**，期望收益由负转正：
```
E = (0.53) × 0.25 - 1.0 × 0.75 = -0.62（当前，负期望）
E = (1.25) × 0.28 - 1.0 × 0.72 = -0.37（修复后，仍负期望但大幅改善）
```

**实际情况可能需要 A/B 对比跑 1-2 周才能收敛到精准值。**

## 风险与边界条件

### 风险 1：下午窗口不可持续

13:00-14:00 胜率 32.1% 是基于 22 个交易日数据。如果市场环境变化（例如进入持续单边下跌或长期横盘），下午窗口可能消失。

**缓解：** 加入窗口有效性检测。如果连续 5 个交易日 13:00-14:00 胜率低于 15%，自动降级下午加成。

### 风险 2：时间止损增加摩擦成本

每 3 个时隙（30 分钟）强制减仓，如果行情刚好在第 4 个时隙反转，就被提前止损了。

**缓解：** 初始 3 slot 参数保守，可根据回测放宽到 4-5 slots。实盘可先用 4 slots。

### 风险 3：回撤止盈 0.3% 过于敏感

0.3% 回撤是 603305 日常波动量级。如果是+1% 浮盈后正常回撤 0.3% 就大量减仓，可能被洗出去。

**缓解：** 如果某时隙有强信号（涨幅≥1.2%），当回撤止盈触发时只减 30%，不是 50%。

### 风险 4：与现有止盈逻辑叠加

新规则（回撤止盈 0.3%、时间止损）和现有规则（浮盈止盈、短线止损）同时生效，需要确定优先级。

**优先级建议：** 时间止损 > 回撤止盈 > 浮盈止盈 > 信号调仓。时间止损最严格，信号调仓最宽松。

## 总结

**对 DeepSeek 方案的总体评价：** 核心思想（时间作为风控变量、不对称持仓）方向正确，但 120 秒执行分辨率在 Kronos 现有的 10 分钟执行框架下无法实现。

**OpenClaw 优化方案：** 将 DeepSeek 思想适配到 10 分钟分辨率，用 `simulate_rules_603305.json` 的参数体系表达，改动量约 5 个参数区块，主要改动为：

1. **时间止损** → 连续 3 个时隙无反转则强制减仓
2. **早盘禁仓** → 09:30-10:00 禁止加仓
3. **止盈降级** → 止盈起点从 3.0% 降至 1.2%（影子策略已验证）
4. **回撤保本** → 浮盈后回撤 0.3% 即减半
5. **下午加成** → 13:00-14:00 窗口加仓幅度 +10%/+15%

方向胜率不会改善（48%），但盈亏比有望从 **0.53 修复到 1.0-1.5**，使策略期望收益由负转正。
