## 10.2 策略比较

### 比较方法

基于 19 个 A/B 档交易日（20260507-20260605），重建价格序列后，对每一时隙模拟各候选策略的信号检测→确认→执行流程。使用 ChatGPTP 建议的阈值（bull ≥ +0.6%, bear ≤ -1.2% vs 昨收），模拟成交后下一时隙的收益。

注意：以下为**方向预测胜率（信号发出后下一时隙的方向对不对）**，非完整价差交易胜率。

### 核心比较表

| 指标 | Baseline (现有) | Candidate-A (两次确认) | Candidate-AZT (+早盘保护) |
|------|:---------------:|:---------------------:|:------------------------:|
| **交易次数** | 324 | **317** ⬇2% | **297** ⬇8% |
| **方向胜率** | 40.7% | **41.6%** ▲| 39.7% ▼ |
| **平均 PnL** | +0.004% | +0.001% | +0.002% |
| **平均盈利** | +0.341% | +0.333% | +0.331% |
| **平均亏损** | -0.256% | -0.253% | -0.236% |
| **盈亏比** | 1.33 | 1.32 | **1.40** ▲ |
| **交易减少幅度** | — | 2.2% | **8.3%** |

### 数据解读

**Candidate-A（两次确认）：**
- 信号被确认率：**91.6%**（324 baseline 中 297 被确认）
- 只过滤掉了 7 个信号（2.2%），其中全部是错误信号
- 胜率提升：40.7% → 41.6%（+0.9%）
- **效果微弱**：因为"价格 vs 昨收"信号是粘性信号，在 10 分钟分辨率下一旦触发通常会持续多个时隙。两次确认机制在实践上几乎等价于单次触发。

**Candidate-AZT（+早盘保护）：**
- 早盘 09:30-09:50 共 37 个时隙被屏蔽（7.6% 的总体时隙）
- 交易次数减少 8.3%（表面减少更多，因为早盘也是低频信号区）
- 盈亏比提升：1.33 → **1.40**（+5.3%，单独受益于移除早盘亏损交易）
- 但方向胜率下降：40.7% → 39.7%（因为早盘的信号有些也是对的）

### ChatGPTP 建议的 Candidate-AZ

ChatGPTP 推荐的 AZ（两次确认 + cross-zero 限制 + cooldown）：

**cross-zero 限制的可行性：** 22 个交易日仅发生 **5 次 cross-zero**，0 次 whipsaw。cross-zero 限制在实盘数据中几乎不会触发，对胜率的影响可以忽略。

**cooldown 限制的可行性：** 同上，发生频率太低。

### 核心发现

**ChatGPTP 的方向胜率数据与现有策略基本一致（~40%），两次确认机制在粘性信号特征下不产生实质提升。**

但 ChatGPTP 的正确贡献在于**二次确认并不增加胜率**——真正有价值的是 ChatGPTP 提出的 **factor_score 体系**（将 r10 + r20 + 连续方向 + 区间位置组合成一个综合分数），这可能比简单的"两次确认同一方向"更有区分度。

## 10.3 评审输出

### 1. 哪个候选方案最值得进入 observer-only 开发

**推荐：candidate-A（优先）→ AZT（后续可选）**

- Candidate-A（两次确认）：改动最小，无副作用的信噪比滤波器。确认后胜率小幅提升 0.9%，虽不惊艳但也不损害系统。
- Candidate-AZT：早盘保护可提升盈亏比（1.33 → 1.40），但方向胜率略有下降，建议在 candidate-A 稳定运行一段时间后再决定是否叠加。

**不推荐 AZ（cross-zero 限制）作为独立候选**——cross-zero 发生频率太低，该限制几乎不会触发。

### 2. 哪些规则应保留

| 规则 | 保留理由 |
|------|----------|
| ✅ **两次确认开仓** | 无副作用的信号滤波器，虽提升不大但不会损害系统 |
| ✅ **factor_score 体系** | **这是 ChatGPT 方案中唯一有独立价值的贡献**——比单纯"价格 vs 昨收"信号具有更高的信息含量 |
| ✅ **完整价差交易台账 (round_trip_trade_ledger)** | 策略评估基础设施，推荐单独推进 |
| ✅ **尾盘保护 (14:30 后不新增)** | 与 DeepSeek 方案不冲突，可独立采用 |

### 3. 哪些规则应暂缓

| 规则 | 暂缓理由 |
|------|----------|
| ⏸️ **早盘全面禁止（11:00 前只减不加）** | DeepSeek 方案是全面禁仓，ChatGPT 是 09:30-09:50 降敏感度。两方案取折中：建议先用 ChatGPT 的"09:30-09:50 只观察"验证 1-2 周，再决定是否升级到 DeepSeek 的全面禁仓 |
| ⏸️ **cross-zero 严格限制** | 在 603305 上 cross-zero 频率极低（22 天 5 次），此规则几乎不会触发。建议仅做监控统计，不实施硬限制 |
| ⏸️ **cooldown 机制** | 同 cross-zero |

### 4. 哪些阈值需要回测或 shadow 验证

| 阈值 | 验证需求 | 优先级 |
|------|----------|--------|
| factor_score ≥ ±60 → 记录 pending | 需要实盘计算统计分布 | P1 |
| factor_score 各因子权重（25/20/20/15/10/10%） | 无数据支撑，**纯理论推导** | P2 |
| 止盈价差 0.8%/1.5% | 当前平均盈利 +0.34%，1.5% 可能过高 | P2 |
| 减仓信号衰减阈值（+30 到 -30 减仓） | 需要先建立 factor_score 的统计分布 | P1 |

### 5. 哪些指标必须新增

ChatGPT 提出的完整价差交易台账（round_trip_trade_ledger）值得实施，但**这是一个系统基础设施升级，不是一个策略参数调整**。

建议新增的指标：

| 指标 | 可行性 | 实施方式 |
|------|--------|----------|
| round_trip_trade_ledger | ✅ 可行 | 在 simulate_position 中增加状态机，记录开仓→平仓事件 |
| profit_factor | ✅ 取决于台账 | 台账就绪后可计算 |
| cost_to_gross_pnl | ✅ 成本数据已有 | 从 cum_cost 即可计算 |
| max_drawdown | ✅ 已有 | close_snapshot 和 report 中已有 |

### 6. 是否建议新增 v1.2-shortspread-shadow

**建议：可以新增，但需要先明确优先级。**

| 理由 | 说明 |
|------|------|
| ✅ 6 项因子评分中 5 项可计算（VWAP 用"近似均价"替代） | 当前数据源支持 |
| ✅ 方向胜率从 40.7% → 41.6% | 轻微正向改进 |
| ⚠️ 二次确认机制在粘性信号下改善有限 | 胜率提升 < 1%，不等于零但不大 |
| ⚠️ Cross-zero/coolown 限制几乎不会触发 | 为 603305 定制的此部分效率低 |
| ❌ Factor_score 权重体系是纯理论，无数据支持 | 实盘运行后才能确定有效权重 |

**如果新增，建议只实现"factor_score + 两次确认"，去掉 cross-zero/coolown 限制。**

### 7. 是否建议保留现有主策略不变

**✅ 强烈建议保留现有主策略不变。**

ChatGPT 方案中的任何候选（A/AZ/AZT）都**不能替换当前主策略**。方向胜率改善极微（+0.9%），远未达到替换门槛。

现有主策略 + DeepSeek 参数优化（时间止损、止盈降级、回撤保本）+ ChatGPT candidate-A（两次确认 shadow），三者并行，是一个合理结构。

### 8. 是否建议继续观察 v1.1-shadow

**✅ 建议继续观察。**

v1.1-shadow 在 81% 的时点优于主策略，是目前最接近"可替代"的候选。ChatGPT 的 candidate-A 不应优先于 v1.1-shadow 升级。

建议优先级：
```
v1.1-shadow 观察（继续） > DeepSeek 参数优化（等待用户同意） > ChatGPT candidate-A（observer-only 开发）
```

### 9. 是否允许任何参数切换

**❌ 不允许。**

当前没有任何候选方案（candidate-A/AZ/AZT）的数据支持自动参数切换。

### 10. 下一步最小开发任务

根据 ChatGPT 的方案和评审结论，下一步最小任务为：

```
1. （用户授权后）修改 simulate_rules_603305.json 实现 DeepSeek 5 项参数优化
2. （并行）在观察者层实现 factor_score 计算脚本（不修改主策略，只输出统计）
3. （观察）统计 factor_score 的实际分布，验证 ±60 阈值是否合理
4. 基于 factor_score 统计分布，再决定是否实现 candidate-A 的两次确认逻辑
5. 不实施 cross-zero/coolown（数据不支持）
```
